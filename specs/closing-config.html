⚙️ Spec: Cấu Hình Kết Chuyển Cuối Kỳ (Closing Config)

Thiết kế cho dev hiểu tại sao cần cấu hình, cách cấu hình hoạt động, và cách code CRUD + closing service.

🎯 Tại sao cần cấu hình kết chuyển?

Theo Điều 79, 80 TT 99/2025/TT-BTC, cuối mỗi kỳ kế toán (tháng, quý, năm), doanh nghiệp phải thực hiện kết chuyển để xác định kết quả kinh doanh (lãi/lỗ).

Quy trình chuẩn:

  1. Kết chuyển Doanh thu (TK 511, 711) → ghi Có TK 911
  2. Kết chuyển Chi phí (TK 632, 641, 642, 811) → ghi Nợ TK 911
  3. Kết chuyển Lãi/Lỗ từ TK 911 → TK 421 (Lợi nhuận chưa phân phối)
💡 Tại sao cần bảng cấu hình riêng?
Vì mỗi doanh nghiệp có hệ thống tài khoản khác nhau (có thể dùng TK 5111, 5112, 5113... hoặc có thêm TK 711 - thu nhập khác). Thay vì hardcode trong code, chúng ta lưu các rule vào closing_rules để kế toán tự cấu hình mà không cần sửa code.

📖 Cách hoạt động của cấu hình

Mỗi rule = 1 bút toán kết chuyển. Bao gồm: TK Nợ, TK Có, loại rule (revenue/cost/profit), diễn giải, thứ tự thực hiện.

3 loại RuleType quan trọng:

revenue (Doanh thu)
TK Nợ = 511/711
TK Có = 911
Chuyển toàn bộ doanh thu trong kỳ vào TK 911.
cost (Chi phí)
TK Nợ = 911
TK Có = 632/641/642/811
Chuyển toàn bộ chi phí trong kỳ vào TK 911.
profit (Lãi/Lỗ)
TK Nợ = 911, TK Có = 421 (nếu lãi)
TK Nợ = 421, TK Có = 911 (nếu lỗ)
Chuyển KQKD về lợi nhuận chưa PP.

Thứ tự thực hiện (SortOrder): RẤT QUAN TRỌNG. Phải KC doanh thu trước → KC chi phí → KC lãi/lỗ sau cùng. Nếu sai thứ tự, số dư TK 911 sẽ không chính xác.

Khi closing service chạy: Hệ thống SELECT tất cả rule từ closing_rules ORDER BY SortOrder ASC → tính số tiền thực tế từ trial_balance hoặc gl → tạo closing_voucher + accounting_entries → INSERT journal, ledger → UPSERT trial_balance.

📥 Thao tác CRUD

Sửa ruleclosing_rules UPDATECập nhật TK, loại, thứ tự. Cảnh báo nếu thay đổi SortOrder làm sai thứ tự revenue → cost → profit.
Xóa ruleclosing_rules DELETEXóa rule. Lưu ý: nếu xóa rule revenue hoặc profit, kết chuyển sẽ KHÔNG đầy đủ → báo cáo KQKD sai.

🌐 API Endpoints

GET    /api/closing-rules                    → List all rules (sorted by SortOrder)
POST   /api/closing-rules                    → Create new rule
PUT    /api/closing-rules/:id                → Update rule
DELETE /api/closing-rules/:id                → Delete rule
PATCH  /api/closing-rules/reorder           → Body: [{ruleId, sortOrder}...] — cập nhật hàng loạt SortOrder

// Ví dụ POST body:
{ "debitAccountId":"acc-511", "creditAccountId":"acc-911", "ruleType":"revenue", "description":"KC Doanh thu bán hàng", "sortOrder":1 }

🔗 Liên kết với Closing Service

Khi người dùng nhấn "Chạy Kết Chuyển" trên closing.html, service sẽ:

1
SELECT * FROM closing_rules ORDER BY SortOrder ASC
2
Với mỗi rule, tính số tiền thực tế của kỳ từ trial_balance:
• RuleType=revenue → lấy SUM(MovementCredit) của TK Nợ
• RuleType=cost → lấy SUM(MovementDebit) của TK Có
• RuleType=profit → tính NetProfit = TotalRevenue - TotalCost
3
INSERT closing_vouchers + closing_voucher_details + accounting_entries (SourceDetailType=4) → INSERT journal, ledger → UPSERT trial_balance.
